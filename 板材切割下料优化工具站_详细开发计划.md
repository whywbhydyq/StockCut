# 板材切割 / 一维二维下料优化排版工具站详细开发计划

版本：v1.0  
日期：2026-05-24  
项目代号：Cut Layout Optimizer  
建议站点形态：英文独立免费工具站，后续可扩展中文页面  
开发原则：先做可上线、可真实使用、可验证搜索和交互信号的 MVP，不做企业级排产 SaaS。

---

## 0. 执行结论

本项目第一版应开发为一个**前端优先、本地计算、无需登录、可打印、可导出、适合 AdSense 与 affiliate 的免费工具站**。

第一版核心产品骨架：

1. **Sheet Cutting Optimizer**：二维板材矩形下料优化，适用于 plywood、MDF、melamine、acrylic、glass、metal sheet。
2. **Linear Cutting Optimizer**：一维长料/型材下料优化，适用于 lumber、steel tube、aluminum extrusion、PVC pipe、rebar。
3. **Saw Kerf Calculator**：锯缝补偿与可行性解释，用于承接锯缝长尾词，也作为核心工具的信任入口。
4. **Printable Cutting Layout**：结果图、零件标签、出材率、废料区、切割步骤、PDF/打印输出。
5. **Preset SEO Pages**：材料、尺寸、项目、型材等长尾页面复用真实工具模块，而不是生成空模板文章。

第一版必须做准：

- 单位解析。
- 锯缝 kerf。
- 2D 矩形排版。
- 1D 长料优化。
- 表格输入与粘贴导入。
- 结果图。
- 打印 / PDF。
- CSV / JSON 导出。
- localStorage 草稿保存。
- 错误提示。
- 移动端基础可用。
- sitemap、robots、canonical、结构化数据、隐私与免责声明。
- Vercel `ignoreCommand` 构建保护。

第一版坚决不做：

- 账号系统。
- 云端保存。
- CNC 后处理。
- DXF。
- G-code。
- 异形 nesting。
- 圆形、三角形、多边形零件。
- 角度切割。
- 企业库存。
- 订单排产。
- 自动报价系统。
- 团队协作。
- AI 柜体设计。

一句话执行标准：

> 先把 “4×8 plywood + kerf + label + printable layout + table paste + local/export save” 做准，再扩展其他高级功能。

## 1. 项目目标

### 1.1 商业目标

本项目不是先做付费 SaaS，而是先验证免费工具站模型：

- 通过明确工具词获取搜索流量。
- 通过可交互工具提高停留和复用。
- 通过 AdSense 承接展示广告收入。
- 通过 affiliate 承接材料、锯片、电锯、夹具、卷尺、木工图纸、课程、服务器/工具类产品等商业邻接。
- 通过 90 天数据判断是否继续扩展。

### 1.2 产品目标

用户输入 stock 与 part list 后，系统应能输出：

- 每张板或每根长料的切割方案。
- 每个零件的位置、标签和尺寸。
- 使用原料数量。
- 出材率和废料率。
- 剩余废料清单。
- 无法放入的零件与原因。
- 可打印 / PDF 的 shop-ready 下料单。

### 1.3 技术目标

- 核心计算在浏览器本地完成。
- 不上传用户 cut list。
- 不要求登录。
- TypeScript 类型约束清晰。
- 算法模块可替换。
- 单位系统可测试。
- SEO 页面可扩展。
- 移动端可用。
- 构建部署可控，避免连续提交浪费 Vercel 构建额度。

## 5. 算法计划

### 5.1 算法原则

第一版文案和代码都不承诺 mathematically optimal。应使用：

- optimized layout。
- practical cutting plan。
- kerf-aware layout。
- guillotine-friendly layout。

避免使用：

- perfect optimization。
- guaranteed optimal。
- industrial-grade nesting。
- CNC-ready output。

### 5.2 2D 算法路线

推荐先采用“可替换引擎接口”。

```ts
export interface SheetPackingEngine {
  name: string
  optimize(input: SheetOptimizationInput): SheetOptimizationResult
}
```

MVP 可选两种实现。

#### 方案 A：自研 Guillotine Engine

优点：

- 更符合实际直线切割。
- 容易生成 cut sequence。
- 容易解释。
- 可控。

缺点：

- 利用率可能不如 MaxRects。
- 对复杂矩形组合效果一般。

#### 方案 B：MaxRects Engine

优点：

- 利用率通常更好。
- 适合矩形 packing。
- 可快速得到可用结果。

缺点：

- 原始算法不直接等于真实锯路。
- cut sequence 需要额外生成。
- 如果作为黑盒，调试困难。

#### 推荐决策

MVP 采用：

- 首版实现简单 `GuillotineEngine`，保证可解释。
- 保留 `SheetPackingEngine` 接口，后续可加入 `MaxRectsEngine` 做 alternative layout。

### 5.3 2D 排版处理规则

1. 展开 quantity 为 part instances。
2. 按面积降序排序。
3. 对每个 part instance，尝试放入已打开 sheet。
4. 若放不下，打开新 sheet。
5. 若超过 stock quantity，进入 unplaced。
6. 每个 placement 的可用尺寸需考虑 kerf。
7. 如果 allowRotation = true，尝试原方向与旋转方向。
8. 如果 part 在无 kerf 时可放、加 kerf 后不可放，返回 `KERF_CAUSES_OVERFLOW`。
9. 如果禁止旋转时不可放、允许旋转可放，返回 `ROTATION_LOCKED_WOULD_FIT_IF_ROTATED`。
10. 每次优化输出 durationMs 和 algorithm name。

### 5.4 Kerf 处理

MVP 简化规则：

- 2D 中把 kerf 视为零件之间的最小间隔。
- 零件与 stock 外边界之间不额外扣 kerf，除非用户设置 trim。
- 1D 中每两个连续 cut 之间扣一次 kerf。
- 最后一段后不扣 kerf。

说明文案必须解释：

- Kerf 是 saw blade 移除的材料宽度。
- 如果从一张板中切多个零件，中间切割会消耗材料。
- 48 in 宽板不能简单切出两个完整 24 in 零件，因为中间切割会消耗 kerf。

### 5.5 1D 算法路线

MVP 使用 Best-Fit Decreasing。

流程：

1. 展开 cuts。
2. 按 length 降序排序。
3. 依次放入剩余空间最接近的 stock。
4. 已有 stock 放不下时打开新 stock。
5. 若所有 stock 用完仍放不下，进入 unplacedCuts。
6. 每根 stock 内计算 cut total、kerf total、waste。

伪代码：

```ts
for cut in cutsSortedByLengthDesc:
  bestStock = null
  bestRemaining = Infinity

  for stock in openedStocks:
    required = cut.length + (stock.cuts.length > 0 ? kerf : 0)
    remainingAfter = stock.remaining - required
    if remainingAfter >= 0 and remainingAfter < bestRemaining:
      bestStock = stock
      bestRemaining = remainingAfter

  if bestStock:
    place cut into bestStock
  else if unopened stock exists:
    open new stock and place cut
  else:
    mark cut unplaced
```

## 11. Vercel / GitHub 工程基建计划

### 11.1 Vercel ignoreCommand

开发初始阶段必须检查是否已有 `vercel.json`。

如果不存在，新增：

```json
{
  "ignoreCommand": "node scripts/skip-old-vercel-builds.mjs"
}
```

如果已存在 `vercel.json`，只合并 `ignoreCommand`，不覆盖其他配置。

### 11.2 skip-old-vercel-builds 脚本

新增：

```text
scripts/skip-old-vercel-builds.mjs
```

逻辑：

1. 读取 `VERCEL_GIT_COMMIT_SHA`。
2. 读取 `VERCEL_GIT_COMMIT_REF`。
3. 读取 `VERCEL_GIT_REPO_OWNER`。
4. 读取 `VERCEL_GIT_REPO_SLUG`。
5. 如果变量不足，为避免误跳过部署，`process.exit(1)`。
6. 查询 GitHub 当前分支最新 commit。
7. 如果当前 Vercel 构建 commit 不是最新 commit，`process.exit(0)` 跳过构建。
8. 如果当前 commit 是最新 commit，`process.exit(1)` 继续构建。
9. 如果检查失败，`process.exit(1)` 继续构建。

### 11.3 提交规范

代码修改应按阶段批量提交：

- 同一阶段先完成全部改动。
- 本地或远端检查通过后再提交。
- 使用 GitHub API 时，优先一次性完成 `create_blob → create_tree → create_commit → update_ref`。
- 避免每改一个文件提交一次。
- 如果实在无法批量提交，才允许小步提交。

### 11.4 状态区分

每次部署后必须区分：

1. GitHub 是否提交到目标分支。
2. Vercel 是否触发构建。
3. Production 是否实际部署最新 commit。
4. 若触发 build rate limit，不要继续堆无意义提交。
5. 若部署失败，先查构建日志，再决定是否提交修复。

## 16. 风险与应对

### 16.1 算法可信度风险

风险：排版看起来省料，但实际不好切或不准确。  
应对：

- 不承诺全局最优。
- 优先 guillotine-friendly。
- 显示 kerf included。
- 输出 warnings。
- 后续增加 alternative layouts。

### 16.2 单位解析风险

风险：fractional inch / metric 混用导致错误。  
应对：

- 单位模块优先开发。
- 大量测试。
- 字段级错误。
- 内部统一整数单位。

### 16.3 功能膨胀风险

风险：做成企业 SaaS，无法快速上线。  
应对：

- 严格遵守不做范围。
- DXF、CNC、账号、库存全部延后。
- 以 90 天数据决定扩展。

### 16.4 SEO 薄内容风险

风险：长尾页只是模板替换，质量低。  
应对：

- 每页绑定真实预设。
- 每页有不同示例和 FAQ。
- 控制首批页面数量。
- 先做 10–15 个，不批量堆 100 个。

### 16.5 AdSense 干扰操作风险

风险：广告影响工具使用或触发误点。  
应对：

- 广告位写入验收标准。
- 输入区和按钮附近禁放。
- 移动端单独检查。

## 20. 关键提醒

这个项目的竞争力不是“公式复杂”，而是“真实工作流完整”。

用户真正要的是：

- 我有材料。
- 我有尺寸。
- 我不想浪费。
- 我怕 kerf 算错。
- 我想直接打印去切。
- 我下次还会再用。

因此，开发中所有取舍都应服务于这条主线。任何不能改善“输入真实 cut list → 生成可信 layout → 打印执行”的功能，都不应进入第一版。
